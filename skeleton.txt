Vichan architecture findings from live vichan site:

1) Runtime bootstrap and configuration precedence
- Entry scripts include inc/bootstrap.php, then rely on inc/functions.php load-time initialization.
- loadConfig() in inc/functions.php builds effective config in this order:
  a) inc/config.php (defaults)
  b) inc/instance-config.php (instance overrides; includes inc/secrets.php)
  c) board-level config.php when a board is opened.
- loadConfig() also derives computed paths/URLs and sets many fallback defaults.
- Board opening is explicit via openBoard(uri), which loads board metadata and reapplies config overrides for that board.

2) Core entrypoints
- post.php: write path for delete/report/post/appeal and NNTP ingress.
- mod.php: moderation router and dispatcher.
- report.php: report form page renderer.
- search.php: search UI and query execution with rate limits.
- smart_build.php + inc/route.php + inc/controller.php: on-demand static rebuild helper mode.
- b.php: banner redirect utility.

3) Frontend posting flow (browser)
- templates/post_form.html renders <form ... onsubmit="return doPost(this);"> with multipart data and board/thread hidden fields.
- templates/main.js defines doPost(form): persists name/email/password in localStorage, stores body draft, and only allows submit if body or file/url exists.
- rememberStuff() in templates/main.js restores saved fields/drafts and clears successful draft state using cookie signaling set by post.php.
- Optional ajax posting behavior is layered by js/ajax.js (appends json_response=1 and posts via FormData) if enabled by additional javascript settings.

4) Post pipeline in post.php (high-level sequence)
- Branches by intent: deletion, report creation, posting, ban appeal.
- Posting branch validates board/thread context, board lock, anti-spam question, ban status, captcha provider flow, referer, DNSBL, and mod permissions.
- Normalizes identity fields (name/email/subject/password), computes post mode (OP/reply), checks body/file constraints and flood/limits.
- Processes flags:
  - country_flags via GeoIP modifier injection.
  - user_flag and multiple_flags parsing/validation into tinyboard body modifiers.
- Runs markup/filter pipeline and computes cite tracking.
- Media path validates extension, MIME/image sanity, dimensions, thumbnail generation, optional EXIF strip/orientation, OCR, duplicate detection.
- Inserts post row via post(array) in inc/functions.php, then writes cites/flood metadata and bumps thread when applicable.
- Rebuild sequence: buildThread(thread_id), buildIndex(), and theme rebuild triggers.
- Responds with redirect or JSON payload; sets JS cookie map to clear local draft cache on success.

5) Data writes and schema shape relevant to pipelines
- Posts are inserted into per-board tables named posts_<board_uri> (dynamic table naming through prepare/query helpers).
- Files are stored as JSON in posts row (files, num_files, filehash), not as normalized child table in current PHP code.
- Reports stored in reports table.
- Feedback stored in feedback; moderator notes in feedback_comments.
- Moderation/auth data in mods, modlogs, pms, bans, and related tables.

6) Rendering pipeline
- Domain display objects are Post and Thread classes in inc/display.php.
- Post/Thread build() render Twig templates via Element(...) calls.
- buildThread(id) in inc/functions.php fetches OP+replies, builds HTML thread pages, and optional JSON API thread payload.
- index(page) builds index payload with thread previews and omitted counters.
- buildIndex() writes board pages and API pages, catalog/threads JSON, and purges stale pages.

7) Build strategy and deferred generation
- generation_strategy() delegates to strategy callbacks (immediate, smart/defer, build_on_load).
- Deferred mode pushes jobs to queue backend; smart_build mode can regenerate on demand when a route is hit.
- smart_build.php uses inc/route.php patterns and inc/controller.php handlers (sb_board/sb_thread/sb_api/etc) to regenerate missing or stale static artifacts.

8) Moderation routing and security model
- mod.php maps regex-like route patterns to mod_* handlers.
- secure and secure_POST route wrappers enforce CSRF token checks (make_secure_link_token).
- check_login() validates moderator cookie session and board-scoped permissions are enforced by hasPermission(...) checks in handlers.

9) Report and feedback pipeline specifics
- report.php renders reporting form and optional captcha.
- post.php handles report submission and inserts rows into reports.
- Feedback public submission is theme-driven:
  - templates/themes/feedback/theme.php emits output index.php at configured path.
  - generated feedback page inserts into feedback table, optionally storing subnet or 0.0.0.0 based on feedback_store_ip.
- Moderator feedback queue lives under mod.php routes (/feedback, /feedback/<id>/delete, /mark_read, /comment) implemented in inc/mod/pages.php and rendered with templates/mod/feedback*.html.

10) Theme system
- Theme lifecycle is driven by theme settings in DB and rebuild calls in inc/functions/theme.php.
- rebuild_themes(action, board) iterates installed themes and dispatches per-theme build callback.
- Theme code can perform schema migration/install-time checks (example: feedback theme install creates/updates feedback tables).

11) Service and dependency structure
- Context container in inc/context.php wires LogDriver, HttpDriver, captcha query services, and cache driver.
- Most legacy code still uses global config/state and free functions, with selective service-based refactors (captcha/http/logging).
- Database access uses PDO wrappers in inc/database.php with dynamic board table prefix expansion and optional debug instrumentation.

12) Architectural constraints to account for in rewrite
- Heavy reliance on global mutable state (config, board, mod, build_pages).
- Request handlers perform orchestration + business rules + IO in one place (especially post.php).
- Filesystem static generation is a first-class output model, not just view rendering.
- Per-board table strategy in MySQL affects query and migration design.
- Feature toggles and board overrides are pervasive and must be preserved by config precedence logic.

13) Suggested Phoenix decomposition from these findings
- Web layer: controllers/live for post/report/mod/feedback and static endpoints.
- Domain contexts: Posting, Media, Moderation, Reporting, Feedback, Build, Themes, Boards, Search.
- Infra adapters: Repo (PostgreSQL), Storage (local/object), Queue, GeoIP, Captcha, Logging.
- Job layer: thread/index/catalog/theme rebuild workers replacing PHP deferred generation semantics.
